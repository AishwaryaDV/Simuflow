# Part 1 — Bugs

## Critical

### 1. Two shipped templates can never be simulated
The `ai_agent` preset (agent-orchestrator ↔ vector-db / graph-db feedback edges) and `ride_sharing`
preset (ws-gw ↔ matching-svc) contain **cycles**. `ERR_CYCLE` in
`frontend/src/validation/rules.ts:158` is a **blocking** error, so loading either flagship template
and pressing Simulate always shows "Simulation blocked". Affects both the frontend JSONs
(`src/presets/ai_agent.json`, `src/presets/ride_sharing.json`) and the seeded rows in
`backend/supabase/migrations/002_create_presets.sql`.

**Fix options:** make the presets acyclic, downgrade cycles to a warning, or teach the engine to
handle cycles (it already half-tolerates them — `topoSort` in `simulation.worker.ts:161` appends
cyclic nodes at the end, feeding them stale/zero RPS).

### 2. Save can silently overwrite the wrong diagram (data loss)
`diagramStore.currentDiagramId` is never cleared by Reset canvas (`Toolbar.tsx:81`), Clear Canvas in
the templates sidebar, or loading a template (`TemplatesSidebar.tsx:196`). Sequence: open saved
diagram → Reset (or load a template) → build something new → Save → the old diagram is
**overwritten** with unrelated content under its old name (`GraphStore.clearCanvas` at line 188 also
never resets `diagramName`). Only `DiagramsModal → New` resets identity correctly.

**Fix:** reset `diagramStore.currentDiagramId` and `graphStore.diagramName` in every
canvas-replacing path.

### 3. Live config edits never reach the running simulation
The worker supports `UPDATE_TOPOLOGY` (`simulation.worker.ts:78`) but nothing ever sends it — the
bridge in `useWorkerBridge.ts` snapshots topology only on START. This breaks the product's core
teaching loop: every template's "Try This" copy ("drop the hit rate to 0.3 and watch the DB
saturate") instructs users to tweak configs mid-run, which does nothing until Stop/Start.

**Fix:** post `UPDATE_TOPOLOGY` from a MobX reaction on `graphStore.topology` while running.

## High

### 4. Opening a saved diagram during a running simulation doesn't stop it
`DiagramStore.loadDiagram` (line 70) swaps the canvas but the worker keeps simulating the old
topology; the HUD keeps ticking and stale runtime state / chaos remains keyed to old node IDs. The
template-load path stops the sim correctly — this path forgot to.

### 5. Three chaos scenarios are no-ops
`APP_GC_PAUSE`, `TRAFFIC_PAYLOAD_EXPLOSION`, and `DATA_CACHE_PERSISTENCE` are in the catalogue and
enum but have no handling in `buildChaosModifier` (`constraintSolver.ts:38`) or `applyEdgeChaos`.
Firing them shows the purple ring and pill but has zero engine effect.

### 6. Drag-to-inject chaos is a dead flow
`ChaosCard` (`NodeLibrary.tsx:151`) sets `application/simuflow-chaos-id` on drag, but
`CanvasPanel.onDrop` (line 273) only reads node/structural types — a chaos card dropped during a
running sim silently does nothing (the "start simulation" toast only fires when idle). The chaos
design doc specifies drop → node-picker inject flow; it was never wired.

### 7. Edge right-click offers node-only scenarios that do nothing
The edge context menu filters by `id.startsWith('NET_')` (`ChaosContextMenu.tsx:217`), which
includes node-targeted scenarios (LB failure, TLS cert, DNS failure, blackhole, NAT, sticky skew,
idle timeout). Fired on an edge they get `targetNodeIds: []`, so the engine ignores them while the
UI shows them active. Should filter by `targetKind === 'edge'`.

### 8. Simulation speed is applied twice
The tick interval scales with speed (`TICK_MS / speed`, worker line 96) **and** per-tick simulated
seconds scale with speed (`TICK_SECS * speed`, line 186) — so simulated time runs at speed² and
`CostStore._accumulate` (which also multiplies by speed on top of the faster ticks) over-accrues:
4× speed spends 16× faster, 0.25× spends 16× slower. The seeded
`outflow.set(n.id, cfg.rps * burst * speed)` at worker line 201 is dead code (overwritten in the
node loop) but misleading. Speed should be applied in exactly one place.

### 9. Deleting a diagram has no confirmation and no error handling
The trash icon in `DiagramsModal.tsx:116` deletes immediately — destructive, irreversible, one
mis-click away, and inconsistent with the confirm dialogs used everywhere else. Failures in delete
**and** in `loadDiagram` are unhandled promise rejections (silent to the user).

### 10. No React error boundary anywhere
Any render error in any component blanks the entire app to a white screen. Worker errors are only
`console.error`'d (`useWorkerBridge.ts:27`) — a crashed simulation looks like a frozen one.
**Fix:** top-level error boundary (plus one around the canvas); surface worker `ERROR` messages as
a toast.

### 11. Blocking sync I/O in async backend endpoints
Every route is `async def` but calls the synchronous Supabase client (and `verify_jwt` does a
network round-trip to Supabase Auth per request). Any request in flight blocks the entire event
loop on the single shared-CPU Fly machine — under concurrent load all requests serialize.
**Fix (simple):** change endpoints to plain `def` so FastAPI runs them in the threadpool.
**Fix (better):** validate JWTs locally with the already-configured but unused
`SUPABASE_JWT_SECRET`, saving a network hop per request.

## Medium

12. **Refresh orphans the open diagram.** localStorage persists only the topology
    (`useLocalStoragePersistence.ts`), not `diagramId`/`diagramName`. After a refresh, Save creates
    a duplicate "Untitled Diagram" instead of updating — testing-plan items 4.2/10.8 fail.
13. **Raw JSON error bodies shown to users.** `api.ts:16` throws `Error(await res.text())`, so
    `SharedViewPage` and `ShareModal` render `{"detail":"Shared diagram not found"}` verbatim.
    Parse `detail` in `request()`.
14. **Opening the Share modal publishes immediately.** `ShareModal`'s `useEffect` fires
    `POST /share` on open — clicking "Share" makes the diagram public before the user confirms
    anything. Consider requiring an explicit "Create link" click.
15. **`requireAuth` drops the intent.** Save/Share/Fork gated behind login discard the pending
    action once the user signs in — they must click again. Also `signInWithGoogle` redirects to
    `window.location.origin`, so signing in from `/shared/:token` dumps the user back at the
    workspace, losing the shared page.
16. **UserButton dropdown never closes on outside click** — `ref` is created (`Toolbar.tsx:16`) but
    no outside-click listener is attached.
17. **Deleting the currently-open diagram leaves its content on the canvas** with
    `currentDiagramId=null`; testing-plan 4.10 expects the canvas to clear.
18. **`totalRequests` is recomputed, not accumulated** (`metricAggregator.ts:66`:
    `tick × current RPS`), so the HUD "Requests" counter jumps down when a burst ends.
    `elapsedSeconds` counts wall-clock regardless of speed — inconsistent with "simulated time"
    cost accrual.
19. **`NumInput` never enforces `max`** on blur (`ConfigPanel.tsx:57` clamps only min) — e.g. LB
    replicas accepts 100 despite `max={32}`.
20. **Backend startup is fragile.** `lifespan` calls `load_presets` unguarded — if Supabase is
    briefly unreachable the app crashes at boot; if the presets migration wasn't applied it
    silently serves `[]` forever (cache never refreshes without a redeploy). `fork_diagram` also
    has a read-modify-write race on `fork_count`.

## Low / polish

- `index.html` title is **"frontend"** — shows in every tab until a shared page overrides it.
- `fmtUsd` renders "$0.500¢" — both currency symbols at once (`MetricsPanel.tsx:108`).
- WAF's blocked traffic counts as `errorRate = blockRate`, permanently inflating the global error
  average; blackhole is documented as "silent" but the engine sets 100% failure.
- ESLint: 58 errors (all `no-explicit-any`) — a lint CI gate would fail today.
- Bundle is a single 1.19 MB chunk (recharts/xyflow/supabase all in one) — worth code-splitting
  `SharedViewPage`.
- `CostPlaceholder` duplicates the entire `BASE_HR` pricing table from `CostStore` — guaranteed
  drift.
- Backend: `python-jose` dependency unused; Dockerfile comment says Railway but deploy target is
  Fly; docs say Python 3.12, Dockerfile uses 3.11; `/health` version hardcoded; CORS allows exactly
  one origin (Vercel preview URLs will fail).
- `frontend/.env` has `VITE_SUPABASE_ANON_KEY=why ` — local auth is broken until the real anon key
  is restored (both `.env`s are correctly gitignored, not committed).
- ChaosToolbar quick-"inject" button path is dead — every catalogue entry has
  `requiresTarget: true`, so every popover row just says "right-click node".

# Part 2 — Parked TODOs and how to close them

### 1. Cost pricing model (the only literal `TODO` — `ConfigPanel.tsx:546`)
The per-node instance/SKU picker ships as a disabled "coming soon" input, rates are hardcoded in
two places, and `docs/pricing-research.md` is a decision doc with every checkbox unanswered.
**To close for deploy:** don't build the pricing API now. Decide "AWS-only, static table" (the
first checkbox), de-duplicate by exporting `BASE_HR` from CostStore, and either enable a simple SKU
dropdown from a static table or remove the disabled input so nothing looks broken. The
pricing-API version stays a post-launch item.

### 2. "Data & Analytics" template category
Empty comment block in `templates/index.ts:292`, an orphaned `data_analytics` icon mapping, and two
preset JSONs (`microservices.json`, `queue_system.json`) that exist in the repo — and are named in
`docs/backend-phases.md` as seed data — but are neither registered as templates nor seeded in the
migration. **To close:** either register the two existing JSONs as templates (cheapest — the
topologies already exist; add entries + details copy, optionally a migration `003` to seed them),
or delete the JSONs and the dead icon mapping.

### 3. Chaos engine completeness
The three no-op scenarios (bug #5) plus the unwired drag-to-inject flow (bug #6).
**To close:** implement GC pause (periodic latency/util spike keyed on `tick % interval`), payload
explosion (latencyAdd + downstream edge throughput cap), cache persistence (cold-start hit-rate
drop); wire `application/simuflow-chaos-id` in `CanvasPanel.onDrop` to open the existing
`ChaosContextMenu` config form at the drop position. **Alternative:** cut the three scenarios from
the catalogue for launch — a no-op chaos button is worse than a missing one.

### 4. Undo/redo
`docs/testing-plan.md` items 2.11–2.12 expect Cmd+Z; there is no undo stack anywhere.
**To close for launch:** a simple snapshot stack in GraphStore (serialize topology on mutation, cap
at ~50) — or explicitly descope and remove it from the test plan.

### 5. Auto-save
Listed as optional in backend-phases Phase 6. Do **not** add auto-save until the diagram-identity
bugs (#2, #12) are fixed — auto-save would amplify the overwrite bug. After that it's a debounced
`PUT` when `isDirty && currentDiagramId`.

### 6. Dead UI scaffolding
`UIStore.theme` (defaults to `'light'` in a dark-only app), `activeModal: 'settings'`, and
`onboardingComplete` are all unused — a theme toggle, settings modal, and onboarding were sketched
and parked. **To close:** delete them for launch; each is a trivial follow-up if wanted later.

### 7. Doc drift (~30 min before deploy)
- backend-phases says "4 presets" — migration seeds 7, frontend registers 8 (incl. blank).
- chaos-design's UI spec is implemented but scenario counts drifted (53 across 6 categories now).
- testing-plan's "Known gaps" section on the cost panel is stale since CostStore landed.

# Part 3 — Deploy readiness

The pipeline itself is in good shape: backend tests green, `tsc` clean, prod build passes,
Dockerfile / fly.toml / vercel.json all present, `.env`s not committed. Remaining gates before
shipping (Fly + Vercel + Supabase):

1. **Fix the blockers users hit in the first five minutes:** template cycles (#1) and
   save-overwrite (#2). Strongly recommended alongside: live config updates (#3) since every
   template's tutorial text depends on it, and the error boundary (#10) since one render bug
   currently blanks the app.
2. **Env wiring:**
   - Vercel: real `VITE_SUPABASE_ANON_KEY`, `VITE_API_URL` → Fly URL, `VITE_SUPABASE_URL`.
   - Fly secrets: `SUPABASE_URL`, `SUPABASE_SERVICE_KEY`, `SUPABASE_JWT_SECRET`,
     `FRONTEND_URL` → exact Vercel origin (CORS is single-origin), `ENV=production`
     (disables `/docs`).
3. **Supabase:** apply both migrations to prod; set the Auth redirect allow-list to the production
   origin (Google OAuth + email confirm + password reset all redirect to
   `window.location.origin`).
4. **Post-deploy smoke test:** `/health` → sign-up email arrives → save → reload → share link in
   incognito → fork.
